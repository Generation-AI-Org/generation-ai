# Depth Perception in Canvas Particle Systems — Research

> Research-only document. Keine Code-Changes. Scope: Techniken identifizieren die echtes raeumliches Volumen fuer Signal-Grid erzeugen, ohne die bereits abgelehnten Effekte (Glow, Blur, Desaturation, Parallax, atmospheric perspective) zu reintroduzieren.

---

## Executive Summary

Der aktuelle Signal-Grid nutzt bereits drei der sechs klassischen monocular depth cues: **relative size**, **occlusion** (via painter-sort), und **motion parallax** (via depth-scaled wander speed). Die subjektive Tiefe ist deshalb "da, aber flach" — die Nodes verhalten sich wie eine 2D-Textur mit Varianz, nicht wie ein 3D-Volumen durch das man hindurchschaut.

Die Vision-Science-Literatur ist hier erstaunlich eindeutig: der staerkste noch nicht eingesetzte Cue ist der **kinetic depth effect / structure-from-motion** (Wallach & O'Connell, 1953). Wenn Nodes sich als Wolke gemeinsam subtil **rotieren** (nicht parallax!), konstruiert das visuelle System zwingend eine 3D-Interpretation, weil eine "rigid 3D rotation" fuer das Hirn simpler ist als die alternative 2D-Erklaerung der Bewegung. Das funktioniert ohne Glow, ohne Blur, ohne Desaturation — rein ueber Bewegungskorrelation.

Der zweitstaerkste Hebel ist eine **echte Perspective-Projektion** statt der aktuellen "zDepth ist nur Skalar fuer size+alpha"-Simulation. Mit `scale = FOV / (FOV + z)` werden Nodes, die sich in Z bewegen, nicht nur groesser/kleiner, sondern sie **driften auch radial vom Vanishing Point weg** wenn sie nach vorne kommen. Das ist der Kern-Trick jeder 3D-Starfield-Demo und ein mathematisch korrektes Pendant zu dem, was Luca bereits intuitiv beschreibt.

**Top-Empfehlung (Hybrid):** Eine unspektakulaere, langsame **Kamera-Y-Rotation (~0.03 rad/s)** ueber eine **echte 3D-Position (x,y,z)** pro Node, projiziert mit `FOV / (FOV + z)`, gesorted nach Z, sonst unveraendert. Keine neuen Visuals — nur ehrlichere Mathematik hinter der bestehenden Optik. Erwartetes Ergebnis: Nodes fliegen sichtbar voreinander her, occlusion flippt wenn sie sich ueberkreuzen, und das Feld fuehlt sich als Volumen statt als Textur an.

## What's Been Tried + Rejected

Damit kuenftige Iterationen nicht im Kreis laufen:

- **Glow / shadowBlur um Nodes** → "zu viel am Leuchten"
- **Blur auf far nodes** → "zu unscharf"
- **Color desaturation toward gray** → "zu Sternenfeld"
- **Parallax / mouse-driven camera shift** → "irritiert"
- **Gradient / soft halos auf Linien** → abgelehnt
- **Atmospheric perspective** (Kontrast + Hue Richtung Fog-Color) → abgelehnt

**Was Luca liked:** crispe weisse/schwarze Dots, bestehende size+alpha-Staffelung, differential wander-speed. Diese Basis bleibt erhalten.

**Implikation:** Jede neue Technik muss *ohne* Zusatzfarbe, *ohne* Weichzeichnung, *ohne* Cursor-gesteuerte Kamera auskommen. Der Zugewinn muss rein geometrisch oder ueber gemeinsame Bewegung entstehen.

## Vision-Science Depth Cues — Applicability Matrix

Alle klassischen monocular + motion-basierten Cues aus der Perzeptionsforschung (Goldstein, Sensation & Perception; Wallach & O'Connell 1953; Rogers & Graham 1979):

| Cue                           | Status          | Notes |
|-------------------------------|-----------------|-------|
| **Occlusion / Interposition** | ✅ implemented  | Painter-sort by zDepth. Staerkster Cue wenn Nodes sich ueberkreuzen. Aktuell unterausgenutzt weil Nodes sich selten ueberlagern (wander ist klein, zDepth-Velocity klein). |
| **Relative size**             | ✅ implemented  | sizeFactor 0.51×..2.00× via pow(z, 0.9). Gut kalibriert. |
| **Motion parallax (lateral)** | ✅ partial      | Differential wander speed 0.35×..1.00×. Echter Parallax (Cursor-gesteuert) abgelehnt. |
| **Kinetic depth effect (SfM)**| ❌ **candidate**| **Gemeinsame** Rotation der gesamten Punktewolke → 3D-Wahrnehmung entsteht zwingend. Zentraler Kandidat. |
| **Linear perspective**        | ❌ **candidate**| Echte `FOV/(FOV+z)`-Projektion statt simulierter Skalar. Nodes driften radial vom Vanishing Point. |
| **Texture gradient**          | ❌ candidate    | Node-Dichte pro Tiefen-Band variabel (mehr Nodes in "Ferne"). Aktuell via pow(0.7)-Bias teilweise da. |
| **Atmospheric perspective**   | 🚫 rejected     | Kontrast + Hue Richtung Fog-Color — explizit abgelehnt. |
| **Shading / shadows**         | 🚫 ~rejected    | Jede Form von shadowBlur oder Halo ist raus. Ein "Phong-lite" ueber radial-gradient fill-Style waere technisch moeglich, aber fuegt Glow-Impression hinzu → verletzt Constraint. |
| **Aerial blur / DoF**         | 🚫 rejected     | Explizit abgelehnt ("zu unscharf"). |
| **Stereopsis / convergence**  | N/A             | Erfordert Stereoscopic Display. Irrelevant fuer Web. |
| **Accommodation**             | N/A             | Monitor-Akkomodation gibt keinen nutzbaren Cue auf Screen. |
| **Familiar size**             | N/A             | Nodes haben keine "echte" Groesse die das Hirn referenzieren koennte. |

**Lesehilfe:** Die drei "candidate"-Zeilen sind der komplette neue Werkzeugkasten. Alles andere ist entweder schon drin oder ausgeschlossen.

### Warum der kinetic depth effect so stark ist

Wallach & O'Connell zeigten 1953: wenn ein rotierender Draht als Schatten auf einen Screen projiziert wird, sehen Betrachter **zwingend** eine 3D-Form. Das Hirn "bevorzugt die rigid-3D-Interpretation gegenueber der non-rigid 2D-Interpretation, weil letztere einfacher ist." ([Michael Bach Demo](https://michaelbach.de/ot/mot-sfm/))

Das Entscheidende: **keine** der rejected techniques ist dafuer noetig. Der Effekt entsteht aus der Bewegungskorrelation allein. Differential motion parallax (Rogers & Graham 1979) arbeitet nach demselben Prinzip und ist als monocular depth cue inzwischen als etwa gleich stark wie binocular disparity dokumentiert ([Journal of Vision 2013](https://jov.arvojournals.org/article.aspx?articleid=2129976)).

## Technique Survey

Jede Technik wird gegen die Constraint-Liste geprueft. Violators werden nicht versteckt sondern markiert.

### Technique 1: Global slow Y-axis rotation (kinetic depth effect)

**Beschreibung:** Die gesamte Nodewolke rotiert langsam um eine vertikale Achse durch das Zentrum des Containers (~0.02-0.04 rad/s, also ~200-300s fuer eine volle Drehung). Jeder Node hat eine echte `z`-Koordinate. Pro Frame wird die Rotation auf (x, z) angewendet, bevor projiziert wird.

**Wie es funktioniert (Math):**
```
// per frame, for each node:
const angle = rotationSpeed * t;          // rotation of camera around Y
const xr = n.x0 * cos(angle) + n.z0 * sin(angle);
const zr = -n.x0 * sin(angle) + n.z0 * cos(angle);
// project:
const scale = FOV / (FOV + zr);
const screenX = cx + xr * scale;
const screenY = cy + n.y0 * scale;
const renderSize = baseRadius * scale;
const renderAlpha = ... * scale  // or via zDepth derived from zr
```
`cx, cy` = Container-Center. `FOV` tunen (groesser = flacher, ~500-1500 typical). `n.x0, y0, z0` = stable 3D-Position, nie mutiert — nur die Rotation erzeugt pro Frame neue (x, z).

**Constraints-Check:**
- ✅ Crispe Dots bleiben (nur Position/Size veraendert, kein Style)
- ✅ Kein Glow, kein Blur
- ✅ Keine Desaturation
- ✅ Keine Mouse-Camera — Rotation ist global, autonom, unabhaengig vom Cursor

**Effort:** M. Die z-Koordinate existiert bereits (als zDepth-Skalar). Sie muss zu einer echten z-Position werden, plus der Rotation-Sweep um die Y-Achse. ~50 Zeilen Math + Projektion. Wander-Logik kann bleiben oder vereinfacht werden.

**Performance:** 576 Nodes × (cos, sin, 2 mul, 1 div) = ~2300 FLOPs/frame. Nichts. Canvas-Budget unveraendert. Ein `cos/sin` pro Frame cachen (nicht pro Node) — Rotation ist global.

**Risk:** Rotation kann bei falscher Speed entweder (a) unbemerkt bleiben (zu langsam) oder (b) "dreht sich" sichtbar werden (zu schnell, wirkt wie ein Karussell). Sweet Spot ist ~1 rad/30-60s. Starker Effekt wenn Nodes teilweise hinter anderen verschwinden und wieder auftauchen.

**Reference:** [Kinetic Depth Effect — Wikipedia](https://en.wikipedia.org/wiki/Kinetic_depth_effect) · [Michael Bach KDE demo](https://michaelbach.de/ot/mot-sfm/) · [Mamboleoo — 3D in 2D canvas](https://www.mamboleoo.be/articles/how-to-render-3d-in-2d-canvas)

### Technique 2: True perspective projection (FOV / z-divide)

**Beschreibung:** Aktuell ist `zDepth` nur ein Skalar fuer sizeFactor und alphaFactor. Die **Bildschirmposition** ist aus der 2D-Wander-Simulation, voellig unabhaengig von Z. In einer echten Projektion haetten Nodes eine 3D-Position und ihre Screen-Position waere `screenX = cx + (x - cx) * FOV / (FOV + z)`. Das heisst: wenn z sich aendert (aktuell macht es das via vz), **verschiebt sich der Node radial vom Zentrum** — Nodes "kommen auf dich zu" oder "gehen weg", nicht nur ueber Groesse, sondern ueber tatsaechliche Bewegung im Bildraum.

**Wie es funktioniert:**
Formel aus George Galanakis (HackerNoon) und dem 3D-Starfield-Tutorial von j0e.io:
```
scale = FOV / (FOV + z)       // scale = 1 bei z=0, <1 bei z>0 (hinten), >1 bei z<0 (vorne)
screenX = cx + (worldX - cx) * scale
screenY = cy + (worldY - cy) * scale
radius  = baseRadius * scale
```
Das ist die Standard-pinhole-camera-Projektion mit focal length = FOV. Der Effekt heisst **radial depth-translation**: je naeher ein Node dem Betrachter ist, desto weiter wird er vom Zentrum weggedrueckt. Far Nodes clumpen sich am Vanishing Point.

**Constraints-Check:**
- ✅ Crispe Dots bleiben
- ✅ Kein Glow/Blur
- ✅ Keine Desaturation
- ✅ Keine Mouse-Camera (FOV ist fix, nicht cursor-driven)

**Effort:** M-L. Erfordert dass Nodes eine echte 3D-Position haben und der wander in 3D passiert. Die aktuelle 2D-Physik muss in (x, y, z)-Physik konvertiert werden. Die bestehenden Konstanten (Z_MIN, Z_MAX, vz) bleiben, aber deren Interpretation wird **geometrisch** statt kosmetisch.

**Performance:** Trivially cheap. 3 Multiplikationen + 1 Division pro Node pro Frame.

**Risk:** Verhaltensaenderung am Hero. Wenn vz bestehen bleibt (Nodes atmen in der Tiefe), bekommt man jetzt echtes "schwebt auf dich zu / von dir weg"-Gefuehl statt nur Groessen-Pulsation. Das kann entweder "whoa Tiefe" oder "das laeuft weg" fuehlen lassen — Z_VEL_MAX eventuell reduzieren (von 0.03 auf 0.01-0.015). Muss getestet werden.

**Hinweis zu Vanishing Point:** Wenn `cx, cy` auf Container-Center gelegt werden, convergen far Nodes am Bildmittelpunkt — das ist **linear perspective** und ein ueber Jahrhunderte belegter depth cue. Alternativ kann der Vanishing Point unten-mittig liegen (Horizont-Feel, wie in einem Landschaftsbild).

**Reference:** [Faking 3D in Creative Coding — George Galanakis](https://medium.com/hackernoon/faking-3d-in-creative-coding-85b6b97eb38b) · [3D Starfield Tutorial — j0e.io](https://www.j0e.io/tutorials/starfield/) · [Dusty Sphere — Rectangle World](http://rectangleworld.com/blog/archives/298) (Formel `projX = fLen/(fLen-z)*x`) · [Gabriel Gambetta — Perspective Projection](https://www.gabrielgambetta.com/computer-graphics-from-scratch/09-perspective-projection.html)

### Technique 3: Depth-driven line-routing through 3D space

**Beschreibung:** Aktuell werden Linien zwischen aktiven Nodes als 2D-Strecke `moveTo(a.x, a.y); lineTo(b.x, b.y)` gezogen. In 3D-Setup sind beide Endpunkte projiziert, also folgt die Linie automatisch der perspektivischen Verjuengung. Aber: wenn ein **drittes**, dazwischenliegendes Node in Z vor der Linie liegt, sollte es die Linie optisch "brechen" — d.h. die Linie sollte dort dunkler oder kurz verdeckt werden.

**Soft-Variante (kein occlusion-clipping):** Linie als **mehrere Segmente** mit pro-Segment-alpha, wobei alpha proportional zur mittleren Z des Segments ist. Eine Linie die von near (z=-50) nach far (z=+300) laeuft hat ein nahes Ende heller, ein fernes duennbler. Mit `ctx.createLinearGradient` kann das einmalig pro Linie gestrokt werden → Linie "verjuengt" sich.

**Constraints-Check:**
- ✅ Crispe Dots unveraendert
- ⚠️ Gradient auf Linien — Luca hat "Gradient / soft halos auf Linien" abgelehnt. Je nach Interpretation: wenn das hiesse "halo" (Weichzeichnung am Rand) → kompatibel. Wenn alpha-gradient auch abgelehnt → inkompatibel. **Klaerungsbedarf.**
- ✅ Kein blur
- ✅ Keine Desaturation

**Effort:** S. `ctx.createLinearGradient(x1, y1, x2, y2)` + addColorStop fuer beide Enden. Ersetzt die bestehende `strokeStyle = rgba(...)`-Zeile.

**Performance:** Vernachlaessigbar. Gradients sind canvas-nativ.

**Risk:** Wenn "Gradient auf Linien" im rejection-Ordner landet → ist diese Technik raus. Falls Luca es so gemeint hat "kein soft bleed, sondern crisp stroke" → alpha-gradient ist moeglicherweise ok weil die Linie weiter 1px scharf bleibt.

**Reference:** [MDN createLinearGradient](https://developer.mozilla.org/en-US/docs/Web/API/CanvasRenderingContext2D/createLinearGradient). Konzept aus klassischer technischer Illustrations-Tradition: near ends bold, far ends faded.

### Technique 4: Occlusion-driven "hidden" nodes (depth test beyond painter)

**Beschreibung:** Painter-Algorithmus malt einfach in Reihenfolge. Ein staerkerer cue entsteht, wenn ein Node **exakt** hinter einem anderen steht → der hintere wird **vom vorderen verdeckt** (nicht nur ueberlagert). Aktuell sind alle Nodes 1-2px — die chance auf echte occlusion bei 576 Nodes ist niedrig. Mit **geringfuegig groesserem radius-range** (z.B. near nodes 3-4px, far 0.5-1px) und echter 3D-Position passiert occlusion haeufiger auf ehrlicher geometrischer Basis.

**Variante:** Ein subtiler "depth-cutout" — ein Pixel vor dem Zeichnen eines near Nodes wird der Hintergrund unter ihm gecleared (kleiner `clearRect` mit `radius+0.5px`), sodass die nahe Node **wirklich** die hintere abschneidet statt sie via alpha-overlay zu mischen. Das gibt dem Gefuehl "harte Kante, Objekt davor".

**Constraints-Check:**
- ✅ Crispe Dots (im Gegenteil — schaerfere Kante)
- ✅ Kein Glow
- ✅ Kein Blur
- ✅ Keine Desaturation
- ✅ Keine Mouse-Camera

**Effort:** S. Einmal groessere Radius-Range konfigurieren, optional eine clearRect vor dem fill pro near-Node.

**Performance:** Null bis leicht negativ. Ein zusaetzlicher clearRect pro Node ist messbar aber bei 576 Nodes < 0.5ms.

**Risk:** Mit hartem occlusion wirken near nodes "solider" — eventuell aber so solide dass das Field sich nicht mehr luftig anfuehlt. Muss balanciert werden (z.B. nur fuer Top-20% z-Nodes anwenden).

**Reference:** Klassisches **interposition / overlap** cue aus Perzeptions-Literatur ([Goldstein's Sensation & Perception Chapter 8](https://courses.washington.edu/psy333/lecture_pdfs/chapter8_DepthSize.pdf)). In der aktuellen Implementation nur via painter-sort genutzt, nicht via echtem overdraw.

### Technique 5: Depth-varying stroke pattern (lines thin-at-far)

**Beschreibung:** Verstaerkung des aktuellen Linien-Depth-Cues. Aktuell: `lineWidth = 0.5 + 1.0 * minZ`. Das heisst eine near-near line ist 1.5px, eine far-far line ist 0.6px. Spread auf `0.3px .. 2.5px` erweitern + optional **dash-pattern** fuer far lines (gestrichelt statt solid) → visuelle Hierarchie.

Alternative: **lineWidth dynamisch pro pixel** via multiple strokes (unsauber) ODER — besser — die Linie als dreieckiges Polygon mit zwei breiten/schmalen Enden zeichnen:
```
ctx.beginPath()
// quad from near-thick to far-thin
ctx.moveTo(aNear_top, ...); lineTo(bFar_top, ...); lineTo(bFar_bottom, ...); lineTo(aNear_bottom, ...)
ctx.fill()
```
Das waere eine echte verjuengte Linie — "Neon-Blade"-Optik ohne Neon.

**Constraints-Check:**
- ✅ Crispe Dots unveraendert
- ✅ Kein Glow
- ✅ Kein Blur
- ✅ Keine Desaturation
- ⚠️ Linie wird zu duenner Polygon-Shape — visuell aber weiterhin stroke-like. Sollte ok sein.

**Effort:** M. Line-rendering von `stroke()` zu `fill()` mit berechneten Eckpunkten umbauen. Normal-Vektor zur Linie berechnen. Overhead: +~4 Punkte pro Linie.

**Performance:** Marginal schlechter (quad-fill vs 1px-stroke) aber bei typischen 5-15 aktiven Linien unbedenklich.

**Risk:** Quad-fills bei schraegen Winkeln koennen bei sub-pixel positionen "aliased" aussehen. Kann mit `ctx.lineJoin = 'round'` + stroke nach fill gefuehlt fixed werden, aber das fuegt wieder einen Halo-Beigeschmack hinzu. Eventuell bleibt stroke die bessere Option, nur mit groesserem lineWidth-spread.

**Reference:** Aktuelles line-rendering im Signal-Grid (`signal-grid.tsx` L766). Spread-Vergroesserung ist low-risk.

### Technique 6: Texture-gradient — depth-dependent node density

**Beschreibung:** Perzeptions-cue `texture gradient` besagt: gleichfoermige Texturen werden in der Ferne **dichter** (weil mehr Elemente pro Pixel). Aktuell ist die Verteilung grid-regelmaessig in XY, aber Z-verteilt via `pow(rawZ, 0.7)` — das gibt mehr nodes im mid/far band. Das ist schon ein schwacher texture-gradient.

**Staerkere Variante:** Nodes in einem **Z-Frustum** statt einem Z-Band verteilen. In Frustum-Geometrie ist die Projektion am Horizont kleiner, also fallen dort mehr Nodes in einen gegebenen Screen-Bereich. Bei echter Perspective-Projektion (Technique 2) ist das **automatisch** der Fall — Nodes weit hinten occupieren weniger Screenraum, also wirkt das Field zum Horizont hin dichter.

**Constraints-Check:**
- ✅ Crispe Dots
- ✅ Kein Glow/Blur/Desaturation
- ✅ Keine Mouse-Camera

**Effort:** S wenn Technique 2 implementiert — es ist ein free-Effekt der Perspektiv-Projektion. Ohne Technique 2: M (manuelle Dichte-Verteilung).

**Performance:** Null Zusatzkosten.

**Risk:** Wenn Vanishing Point am Zentrum liegt und die Dichte dort explodiert, koennte der Center zu "gedraengt" werden. Kann mit leichtem density-taper im Hero-Zentrum abgemildert werden (aber vermutlich automatisch schon durch das bestehende radial-fade-overlay gehandlet).

**Reference:** Texture gradient depth cue — [J. J. Gibson 1950 — The Perception of the Visual World](https://en.wikipedia.org/wiki/Texture_gradient). Webbasierte Demo: [Gabriel Gambetta Computer Graphics from Scratch](https://www.gabrielgambetta.com/computer-graphics-from-scratch/09-perspective-projection.html).

### Technique 7: Subtle differential-rotation bands (multi-axis KDE)

**Beschreibung:** Statt einer globalen Y-Rotation (Technique 1) rotieren **Bands** (z-slices) mit unterschiedlichen Geschwindigkeiten oder Achsen. Z.B. far band rotiert nur Y, mid band rotiert Y + minimal X-tilt, near band rotiert Y etwas schneller. Das ist staerker informativ fuer das Hirn weil es mehrere konsistente rigid-body-Rotationen sieht die nur bei 3D-Struktur simultan valid sind.

**Constraints-Check:**
- ✅ alle constraints ok
- ⚠️ komplex — kann "uneasy" wirken wenn die Bands nicht harmonisch synchron sind

**Effort:** M-L. Erfordert Abstimmung der Rotationsspeeds so dass sie nicht dissonant werden. 3 verschiedene Rotationsmatrizen pro Frame (oder 1 mit z-dependentem Winkel).

**Performance:** trivial (3 sin/cos pro Frame, nicht pro Node).

**Risk:** Hoch. Diese Technik ist elegant in Theorie, aber empirisch schwer in "angenehm" zu tunen. Erstmal Technique 1 in Isolation testen.

**Reference:** [Journal of Vision — Dynamic perspective cues enhance depth from motion parallax](https://jov.arvojournals.org/article.aspx?articleid=2597818) zeigt dass mehrere korrelierte Motion-Cues multiplikativ wirken statt nur additiv.

### Technique 8: Depth-biased breathing (asymmetric Z-oscillation)

**Beschreibung:** Aktuell `vz` wird random nach oben/unten gesetzt. Statt symmetrisch: near nodes oscillieren in Z-Richtung subtil **nach vorne** (kommen auf uns zu), far nodes driften **weg**. Das erzeugt einen permanenten "through-flight"-Loop analog zum klassischen Starfield, aber extrem subtil (0.005 depth/sec). Bei Perspective-Projektion wirkt das wie ein sehr langsamer Camera-Dolly.

**Constraints-Check:**
- ✅ Kein Glow/Blur/Desaturation
- ✅ Keine Mouse-Camera
- ✅ Crispe Dots

**Effort:** S. Z-velocity-Vorzeichen abhaengig von zDepth biasen.

**Performance:** Null.

**Risk:** Bei hoher Amplitude wird es zum Starfield ("zu Sternenfeld"). Extrem subtil halten.

**Reference:** Klassische 3D-Starfield-Technik — [Kirupa Animated 3D Starfield](https://www.kirupa.com/animations/animated_3d_starfield_effect.htm).

---

### Technique 9: Luminance-contrast preservation across depth (brightness cue)

**Beschreibung:** Die Michael-Bach-KDE-Demo erwaehnt explizit: "Der Tiefeneffekt verschwindet wenn Balls und Background nur in Farbe, nicht in Helligkeit, differieren." Das ist kein neuer Cue, aber eine **Verifikationsregel**: fuer den kinetic depth effect muss luminance-contrast zwischen Node und Background da sein. Aktuell `--text` vs `--bg` — das ist luminance-contrast pur. ✅ check.

**Was daraus folgt fuer Technique 1:** Da wir bereits maximalen luminance-contrast haben, wird KDE zuverlaessig triggern. Waere `--text` und `--bg` nur farblich unterschiedlich (gleich hell), wuerde KDE nicht greifen.

**Constraints-Check:** Bereits erfuellt, kein Action-Item.

**Reference:** [Michael Bach SfM Demo](https://michaelbach.de/ot/mot-sfm/) → explizite Warnung zum Luminance-Requirement.

---

### Technique 10: Pulse-as-depth-indicator (activation reveals Z-layer)

**Beschreibung:** Wenn eine line zwischen zwei aktiven Nodes entsteht, **kurz und unmerklich** alle Nodes im gleichen Z-band wie die aktiven Nodes einen minimalen Pulse geben (alpha +0.1, 200ms). Das reveals temporaer die "Ebene" die der Node bewohnt. Staerkt das Connection-Feeling weil eine Linie nicht nur zwischen zwei Punkten ist, sondern kurz deren **Layer** aufleuchten laesst.

**Constraints-Check:**
- ✅ Keine rejected techniques
- ⚠️ Subtile Aenderung der activation-logik — muss sehr dezent sein sonst wirkt es wie ein Halo-Ersatz

**Effort:** M. Z-band-nachbarschafts-lookup + band-pulse logic.

**Performance:** +1 O(n) pass ueber nodes bei activation. Bei ~2-3 activations/sec: vernachlaessigbar.

**Risk:** Kann schnell zu "busy" werden. Sollte opt-in feature sein, nicht default.

**Reference:** Inspiration aus Gestalt connectedness ([Interaction Design Foundation — Gestalt Principles](https://ixdf.org/literature/topics/gestalt-principles)) — wenn eine connection entsteht, soll der Layer als Gruppe sichtbar werden.

## Recommended Hybrid

**Name:** *Volume-through-rotation* — minimal-invasive 3D-Upgrade des Signal-Grids.

**Idee in einem Satz:** Gib jedem Node eine echte 3D-Position, rotiere die Wolke langsam um die Y-Achse, projiziere mit `FOV/(FOV+z)`, sonst alles gleich lassen.

### Visuelles Resultat (erwartet)

- Nodes **gleiten tatsaechlich voreinander her** (occlusion flippt sichtbar wenn sie sich ueberkreuzen)
- Der Vanishing Point im Zentrum wird spuerbar — far nodes clumpen dort, near nodes driften radial nach aussen
- Das Field wirkt als **Wolke mit Tiefe**, nicht als Textur mit Varianz
- Der kinetic depth effect aktiviert sich *zwingend* nach ca. 2-3 Sekunden Betrachtung (Hirn schaltet um)
- Keine neuen visuellen Elemente — dieselben crispen Dots, dieselben Linien, nur in ehrlicherer Mathematik

### Code-Approach-Skizze (pseudocode, kein build)

```ts
// ─── Add per node ──────────────────────────────────────────────
// Statt zDepth-Skalar: echte 3D stable origin
node.x0, node.y0, node.z0      // world position (nie mutiert, only target)
node.vx3, node.vy3, node.vz3   // 3D velocity (replaces 2D vx,vy)

// ─── Constants ─────────────────────────────────────────────────
const FOV = 800                  // focal length in world units
const ROT_SPEED = 0.03           // rad/sec around Y, ~3.5min per full revolution
                                 // Tunable: 0.02 (extrem subtil) bis 0.06 (erkennbar)
const Z_RANGE_WORLD = 400        // world-z spread (deep to shallow)

// ─── Init ──────────────────────────────────────────────────────
// Seed each node's (x0, y0, z0):
//   x0 = grid_x_in_world   (e.g. gridX - cx)
//   y0 = grid_y_in_world   (gridY - cy)
//   z0 = seeded in [-Z_RANGE/2, +Z_RANGE/2] with pow-skew (more far, less near)

// ─── Per frame ─────────────────────────────────────────────────
const angle = ROT_SPEED * t
const cosA = Math.cos(angle)
const sinA = Math.sin(angle)

for (let n of nodes) {
  // 3D wander (optional — can replace existing 2D wander)
  n.x0 += n.vx3 * dt
  n.y0 += n.vy3 * dt
  n.z0 += n.vz3 * dt
  // ... boundary logic in 3D

  // Apply global camera rotation around Y-axis (rotates x,z; y unchanged)
  const xr = n.x0 * cosA + n.z0 * sinA
  const zr = -n.x0 * sinA + n.z0 * cosA
  const yr = n.y0

  // Perspective projection
  const zCam = zr + FOV         // push camera back so FOV/(FOV+z) stays positive
  const scale = FOV / zCam
  n.screenX = cx + xr * scale
  n.screenY = cy + yr * scale
  n.renderSize = baseRadius * scale         // size now driven by scale, not Math.pow
  n.renderAlpha = clamp(scale * 1.0, 0.3, 1.0)   // far = fainter
  n.sortKey = zr                             // for painter sort
}

// Sort by sortKey descending (largest z = farthest = draw first)
nodes.sort((a, b) => b.sortKey - a.sortKey)

// Render exactly as before — just using screenX, screenY, renderSize, renderAlpha
```

### Was bleibt unangetastet

- Crisp circle render-path (Phase E)
- Mini-net activation logic (Phase C)
- Line rendering (Phase F) — Endpunkte sind jetzt `screenX/screenY`, also automatisch perspektivisch
- Theme-Awareness, reduced-motion handling, cleanup logic
- Color resolution — weiter `--text` / `--bg`

### Neue Tuning-Knobs (fuer Luca-UAT)

| Knob | Default | Range | Effekt |
|------|---------|-------|--------|
| `FOV` | 800 | 400-1500 | klein = starke Perspective, gross = fast flach |
| `ROT_SPEED` | 0.03 rad/s | 0 - 0.1 | 0 = statisch, 0.03 = subtil, 0.1 = erkennbare Rotation |
| `Z_RANGE_WORLD` | 400 | 200-800 | Tiefe der Wolke |
| Vanishing Point | center | center / below / above | wo far-nodes clumpen |

### Was *nicht* passiert (constraints-verify)

- ❌ Kein glow, kein shadowBlur, kein halo → fill() bleibt identisch
- ❌ Kein blur → keine filter: blur, kein radial-gradient fill
- ❌ Keine Desaturation → color weiter `--text`, nur alpha via scale
- ❌ Keine Cursor-driven Kamera → Rotation ist global + autonom
- ❌ Kein atmospheric perspective → kein hue/kontrast-shift

### Rollout-Plan (ohne jetzt zu bauen)

1. **Spike** als isoliertes experiment (separater test-route, gleiche container-Groesse) — nur Technique 1+2, gleiche node count, gleiche colors
2. **UAT mit Luca** bei 3 Rotation-Speeds (0.02, 0.04, 0.06) und 3 FOV-Werten
3. Entscheidung: welche Knob-Kombi → production
4. Einbau in `signal-grid.tsx` mit feature-flag fuer rollback, nachdem klar ist welche Knobs
5. **Tight review** ob line-rendering und activation-pulse noch stimmig wirken bei rotierender Basis (hop-delays koennen zum Rotation-Feel falsch sitzen)

### Fallback wenn Hybrid nicht traegt

- Technique 1 ohne Technique 2 (Rotation ohne echte Projektion — Nodes skalieren ohne radial zu driften). Weniger stark, aber weniger riskant.
- Technique 3 solo (groesserer radius-spread + clearRect-occlusion) — low-hanging fruit, 20min work, messbarer Gewinn.
