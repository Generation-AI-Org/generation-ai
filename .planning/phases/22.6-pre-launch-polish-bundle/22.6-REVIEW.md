# Phase 22.6 Track B: Code Review Report

**Reviewed:** 2026-04-25
**Depth:** standard
**Files Reviewed:** 4
**Status:** issues_found (0 critical, 4 warnings, 6 info)

## Summary

Track B (tools-app polish) ist insgesamt sauber implementiert. Alle CTA-`href`s sind hardcodierte String-Literale ohne User-Input-Interpolation — **kein XSS-, kein Open-Redirect-Risiko**. Keine `dangerouslySetInnerHTML`, kein `eval`, keine inline-`<script>`-Tags. Die einzige inline-`style`-Verwendung (`tools-hero.tsx`, CSS-Token-Reference) ist CSP-safe.

Die Hauptbefunde sind **A11y-Lücken am Mobile-Burger** (vom Executor bewusst übersprungen, sollten aber als Tech-Debt dokumentiert sein) sowie **Inkonsistenzen im State-Reset** zwischen Backdrop-Click und Escape-Handler in `HomeLayout.tsx`. Test-Coverage hat eine kleine Lücke: das neue Mobile-Nav (Burger + Panel) hat keinen E2E-Test in `tools-app.spec.ts`.

CSP-Nonce-Safety: keine Verletzungen erkennbar. `motion/react` injiziert dynamic styles via React-State, was mit dem Next-15 + force-dynamic-Pattern aus LEARNINGS.md kompatibel ist.

## Warnings

### WR-01: Mobile-Burger ohne Focus-Management — A11y-Regression

**File:** `apps/tools-app/components/layout/GlobalLayout.tsx:293-380`
**Issue:** Der Mobile-Burger öffnet ein Panel, aber:
- Kein Auto-Focus auf erstes Nav-Item beim Öffnen → Tastatur-User müssen erst durch alle Header-Elemente tabben.
- Kein Escape-Key-Handler → Panel kann nur per Click auf Burger geschlossen werden, nicht via Tastatur.
- Kein Focus-Trap → Tab im offenen Panel verlässt das Panel und landet im Hintergrund-Content (der visuell durch das Panel verdeckt sein kann).
- Kein Click-Outside-Schließen → Panel hat `onClick={(e) => e.stopPropagation()}`, aber der Header darunter clear nur Highlights, schließt nicht.

Der Executor hat Focus-Trap explizit übersprungen, um keine neuen Deps (`focus-trap-react`) einzuführen — das ist eine valide V1-Entscheidung. Escape-Handler und Auto-Focus sind aber ohne neue Deps machbar (≈10 Zeilen `useEffect`).

**Fix:**
```tsx
// In GlobalLayout.tsx, nach den useState-Calls:
useEffect(() => {
  if (!mobileNavOpen) return
  function handleKeyDown(e: KeyboardEvent) {
    if (e.key === 'Escape') setMobileNavOpen(false)
  }
  window.addEventListener('keydown', handleKeyDown)
  return () => window.removeEventListener('keydown', handleKeyDown)
}, [mobileNavOpen])

// Auf dem Panel-Container zusätzlich:
<div
  data-tools-nav="mobile"
  role="dialog"
  aria-modal="false"
  aria-label="Hauptnavigation"
  // … bestehendes className
>
```

Focus-Trap und Auto-Focus können in eine 22.7-Folge-Phase verschoben werden — bitte als Tech-Debt-Eintrag in STATE.md/ROADMAP.md ergänzen.

### WR-02: Inkonsistenter State-Reset zwischen Backdrop-Click und Escape

**File:** `apps/tools-app/components/HomeLayout.tsx:67-73, 99`
**Issue:** Der Escape-Handler resetted `showSearch`, `searchQuery`, `selectedIndex` UND `highlightedSlugs` (Zeilen 67-73). Der Backdrop-Click (Zeile 99) resetted **nur** `showSearch`. Folge: Wenn der User die Suche per Backdrop-Click schließt und dann per ⌘K wieder öffnet, ist die alte Query + Selection noch da. Das ist entweder ein Bug oder gewolltes Verhalten — aber ohne Code-Kommentar nicht erkennbar.

**Fix:**
```tsx
// HomeLayout.tsx, Zeile 96-100:
<div
  className="absolute inset-0 z-50 ..."
  onClick={() => {
    setShowSearch(false)
    setSearchQuery('')
    setSelectedIndex(0)
    setHighlightedSlugs([])
  }}
>
```

Oder besser: einen `closeSearch()`-Helper extrahieren und sowohl in Escape-Handler als auch Backdrop-Click verwenden (DRY).

### WR-03: Mobile-Nav-Tests fehlen komplett in tools-app.spec.ts

**File:** `packages/e2e-tools/tests/tools-app.spec.ts:51-73`
**Issue:** `B-req-4` testet ausschließlich `[data-tools-nav="desktop"]`. Der neue Mobile-Burger (`[data-tools-burger]`) und das Mobile-Panel (`[data-tools-nav="mobile"]`) haben **keine E2E-Coverage**. Bei einem Mobile-Viewport-Run würde nicht auffallen, wenn:
- Der Burger nicht reagiert (onClick broken).
- Das Panel mit den falschen Items rendert.
- `aria-expanded` nicht togglet.
- Die CTAs im Mobile-Panel andere `href`s haben als im Desktop-Header (drift-risk weil Code dupliziert ist, siehe IN-02).

**Fix:** Test ergänzen:
```ts
test('mobile-nav: Burger toggelt Panel mit allen 5 Items + CTAs (B-req-4-mobile)', async ({ page }) => {
  await page.setViewportSize({ width: 375, height: 812 })
  await page.goto(TOOLS_URL)

  const burger = page.locator('[data-tools-burger]')
  await expect(burger).toBeVisible()
  await expect(burger).toHaveAttribute('aria-expanded', 'false')

  await burger.click()
  await expect(burger).toHaveAttribute('aria-expanded', 'true')

  const mobileNav = page.locator('[data-tools-nav="mobile"]')
  await expect(mobileNav).toBeVisible()
  await expect(mobileNav).toContainText(/Events/)
  await expect(mobileNav).toContainText(/Tools/)
  await expect(mobileNav).toContainText(/Community/)
  await expect(mobileNav).toContainText(/Für Partner/)
  await expect(mobileNav).toContainText(/Über uns/)
  // Verify CTAs match desktop hrefs (drift-guard)
  await expect(mobileNav.locator('a[href="https://generation-ai.org/join?utm_source=tools"]')).toBeVisible()
})
```

### WR-04: B-req-5 schwacher Assert (toHaveCount statt toBeVisible)

**File:** `packages/e2e-tools/tests/tools-app.spec.ts:105-106`
**Issue:** Der Test asserted `toHaveCount(1)` für Settings-Link und Signout-Button, weil die Buttons `md:hidden` sind und auf Desktop-Viewport nicht visible wären. Der Kommentar erklärt das korrekt, **aber:** Wenn jemand später die `md:hidden`-Klasse entfernt oder die Buttons auf Desktop sichtbar macht, würde der Test trotzdem grün bleiben — die Regression-Schutzfunktion ist schwächer als nötig.

**Fix:** Viewport explizit auf Mobile setzen und `toBeVisible()` nutzen:
```ts
await page.setViewportSize({ width: 375, height: 812 })
// nach Login:
await expect(page.getByLabel('Einstellungen')).toBeVisible()
await expect(page.getByLabel('Abmelden')).toBeVisible()
```

Das macht den Test deterministisch und schützt gegen beide Drift-Richtungen.

```tsx
] as const
```
Dann in beiden Renders mappen. V1-akzeptabel — Mirror-Header-Pattern ist bereits via Kommentar (Zeilen 4-8) als Tech-Debt markiert ("Future: extract to @genai/ui").

### IN-04: registerOpenSearch ohne useCallback

**File:** `apps/tools-app/components/layout/GlobalLayout.tsx:78-80`
**Issue:** `registerOpenSearch` wird bei jedem `GlobalLayout`-Render neu instanziiert. Der Context-Value-Object (Zeile 84) ist deshalb ebenfalls jedes Render neu → triggert HomeLayout's `useEffect` (`HomeLayout.tsx:52-58`) potenziell öfter als nötig. In der Praxis harmlos, weil die Registration idempotent ist, aber Code-Quality-Smell.
**Fix:**
```tsx
const registerOpenSearch = useCallback((fn: (() => void) | null) => {
  setOpenSearch(() => fn)
}, [])

const searchContextValue = useMemo(
  () => ({ openSearch, registerOpenSearch }),
  [openSearch, registerOpenSearch]
)
```

### IN-05: Inkonsistentes stopPropagation auf Header-Children

**File:** `apps/tools-app/components/layout/GlobalLayout.tsx:89, 91-143, 226-289`
**Issue:** Der `<header>` hat `onClick={() => setHighlightedSlugs([])}`. Manche Children haben `e.stopPropagation()` (Theme-Toggle, Such-Buttons, Burger, Mobile-Panel, Logout-Form), andere nicht (Logo-Link, Desktop-Nav-Items, CTAs, Settings-Link, Legal-Links). Folge: ein Click auf einen Nav-Link clear erst Highlights und navigiert dann — wahrscheinlich gewollt, aber inkonsistent zur Theme-Toggle-Logik. Ohne Code-Kommentar nicht erkennbar, was Absicht ist.
**Fix:** Entweder Kommentar ergänzen ("Highlights werden bei jedem Header-Click bewusst gecleart, außer bei Modal-trigger Buttons") oder `stopPropagation` konsistent auf alle interaktiven Children setzen.

### IN-06: window.location.href statt router.push für Search-Enter

**File:** `apps/tools-app/components/HomeLayout.tsx:89`
**Issue:** `window.location.href = `/${searchedItems[selectedIndex].slug}`` triggert einen Full-Page-Load statt einer SPA-Navigation. Slug kommt aus `ContentItemMeta` (server-controlled, nicht User-Input) → **kein Open-Redirect-Risiko**. Aber: Inkonsistent mit den `<a href={...}>`-Links in Zeilen 125-148, die ebenfalls Full-Page-Loads triggern (browser-default für `<a>`). In tools-app gibt's eh keine SPA-Navigation für Detail-Pages (siehe Settings-Link Zeile 245 nutzt auch `<a>` "to bypass router cache"), also konsistent mit dem Projektpattern.
**Fix:** Akzeptabel zu lassen. Optional: Kommentar ergänzen, warum `window.location.href` statt `router.push` (= "Detail-Pages brauchen Full-Reload um RSC-Cache zu bypassen").
