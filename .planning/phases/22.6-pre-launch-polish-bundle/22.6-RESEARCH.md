## Summary

Phase 22.6 ist ein Zwei-Track-Polish-Bundle: **Track A** baut eine vollständige `/events`-Seite auf `apps/website` mit MDX-Pipeline (Phase 26 Pattern vollständig wiederverwendbar), Modal-Detail, Signup-Gate-Flow und SEO. **Track B** poliert `apps/tools-app` mit Login-Button-Umbau, Hero-Sektion und Nav-Sync zur Hauptdomain.

Der technische Aufwand ist gut einschätzbar: Track A baut auf dem vollständig verifizierten Phase-26-MDX-Pattern auf (`readAllFrontmatter` + wrapper-Funktion + Validator-Callback), mit dem einzigen Unterschied eines Events-spezifischen Frontmatter-Schemas. Track B ändert ausschließlich `GlobalLayout.tsx` und fügt neue Komponenten hinzu — kein Touch an packages/ui.

**Primary recommendation:** Track A zuerst vollständig abschließen (Plans 22.6-01..05), dann Track B (Plans 22.6-06..09). Innerhalb jedes Tracks maximale Wave-Parallelität wo sinnvoll. Die wichtigsten Pitfalls sind: (1) Modal muss `@base-ui/react/dialog` nutzen — es ist bereits installiert, (2) Auth-Check in der Anmelde-Route muss `force-dynamic`-kompatibel sein (CSP-Nonce-Constraint), (3) tools-app GlobalLayout ist `h-screen overflow-hidden` — Hero-Sektion muss scrollbar bleiben.

#### 5. Auth-Check Pattern (Decision A-07)

Der `/events`-Anmelde-Flow braucht Auth-Check. **Empfohlener Ansatz: Client-side Supabase-Session-Check via `useAuth()`-Hook** (analog tools-app `AuthProvider`) — einfacher als Route-Handler und CSP-kompatibel.

Alternative (A-07 "Server-side bevorzugt"): Route-Handler `POST /api/events/register?slug=[slug]` der `getUser()` aufruft und dann redirect zurückgibt. Nachteil: Extra HTTP-Roundtrip. Vorteil: kein Client-Bundle-Import von Supabase auf der Events-Page.

**Praktische Empfehlung für den Planner:** Client-side Check via `@genai/auth` Client-Helper ist ausreichend für diese Conversion-Seite. Der Auth-State ist sowieso nicht SSR-kritisch (die Events sind öffentlich sichtbar, nur der Anmelde-Button verhält sich unterschiedlich). Pattern: `useAuth()` → wenn kein User → `router.push('/join?redirect_after=/events/'+slug)`, wenn User → `window.open(ctaUrl)`. [VERIFIED: /apps/website/components/join/join-form-card.tsx (redirect_after Pattern) + /apps/tools-app/components/AuthProvider.tsx (useAuth Hook)]

## Common Pitfalls

### Pitfall 1: CSP + force-dynamic (KRITISCH)

**Was schiefläuft:** Die events-Seite nutzt `await headers()` für den Nonce (CSP-Pflicht). Wenn die page.tsx statisch gerendert wird (`○`), gibt es beim Prod-Build keinen Nonce → schwarze Seite (LEARNINGS.md Incident).

**Warum es passiert:** Das Root-Layout hat `export const dynamic = "force-dynamic"`. Solange die events-page `await headers()` aufruft (für den Nonce), ist sie automatisch dynamic. Wenn jemand `await headers()` entfernt oder vergisst: Problem.

**Vorbeugung:** Nach Build `ƒ`-Check: `pnpm build` Output für `/events` und `/events/[slug]` muss `ƒ` (dynamic) zeigen, nicht `○`. Smoke-Test nach lokalem Prod-Build. [VERIFIED: LEARNINGS.md + AGENTS.md gelesen]

### Pitfall 2: tools-app h-screen + Hero-Scroll

**Was schiefläuft:** `GlobalLayout.tsx` hat `<div className="flex flex-col h-screen overflow-hidden">`. Wenn die Hero-Sektion außerhalb des `flex-1 overflow-y-auto`-Containers eingefügt wird, scrollt sie nicht mit — der Inhalt wird abgeschnitten.

**Warum es passiert:** Das tools-app Layout ist `h-screen`-fixed mit internem Scroll (Design-Entscheidung für die Tool-Bibliothek). Eine Hero-Sektion verändert das Layout-Schema.

**Vorbeugung:** Hero innerhalb des overflow-y-auto Wrappers in HomeLayout platzieren, nicht außerhalb. Oder: Den overflow-y-auto Wrapper so anpassen, dass er Hero + FilterBar + Grid umfasst. Test: Mobile Scroll-Smoke auf dem Dev-Server. [VERIFIED: GlobalLayout.tsx + HomeLayout.tsx gelesen]

### Pitfall 3: @base-ui/react Dialog vs. Sheet Naming

**Was schiefläuft:** `sheet.tsx` importiert `Dialog as SheetPrimitive from "@base-ui/react/dialog"`. Das Event-Modal soll auch `Dialog` verwenden. Developer baut fälschlich einen eigenen `<dialog>` HTML-Tag oder vergisst die Portal-Wrapping → Overflow-Clipping, Z-Index-Probleme.

**Vorbeugung:** `event-modal.tsx` direkt von `@base-ui/react/dialog` importieren, analog `sheet.tsx` Pattern. `Dialog.Portal` + `Dialog.Backdrop` + `Dialog.Popup` verwenden. Kein nativer `<dialog>` HTML-Tag. [VERIFIED: sheet.tsx gelesen, @base-ui Exports verifiziert]

### Pitfall 4: Event-Slug aus Filename ≠ Frontmatter-Slug

**Was schiefläuft:** `validateEventFrontmatter` muss (analog `validateArticleFrontmatter`) prüfen, dass `frontmatter.slug === fileSlug`. Sonst gibt es Build-time 404s wenn das MDX-File umbenannt wird ohne Frontmatter-Update.

**Vorbeugung:** In `validateEventFrontmatter` explizit prüfen: `if (raw.slug !== fileSlug) throw new Error(...)`. Analog community.ts Zeile 53. [VERIFIED: community.ts gelesen]

### Pitfall 5: tools-app Nav Cross-Domain-Links

**Was schiefläuft:** Im tools-app Nav müssen alle Nicht-tools-app Links als `<a href="...">` (full page load) implementiert werden, NICHT als `<Link>` von Next.js. `Link` ist nur für interne Routen. `target="_blank"` nur wenn explizit gewünscht (B-05: gleicher Tab).

**Vorbeugung:** Alle `generation-ai.org/*`-Links als `<a href="...">` (kein `target="_blank"`). Nur `/login`, `/settings` als Next.js `<Link>`. [VERIFIED: website header.tsx Pattern für External-Links gelesen]

export async function getUpcomingEventsFrom(dir: string): Promise<EventEntry[]> {
  const now = new Date()
  const items = await readAllFrontmatter<EventFrontmatter>({
