### WR-02: `group-hover` im FAQ-Footer-Link ohne `group`-Parent — Arrow-Hover-Effekt ist dead code

**File:** `apps/website/components/sections/kurz-faq-section.tsx:212-222`
**Issue:** Der `<Link>` zu `/about#faq` hat die Klasse `inline-flex` + `hover:text-[var(--accent)]`, aber der `<ArrowRight>` innen nutzt `group-hover:translate-x-0.5`. Der Link selbst ist nicht als `group` markiert. Resultat: Der Arrow bewegt sich beim Hover nie — ein stilles UX-Regression ggue. Offering/Community-Preview, wo das Pattern korrekt ist (`group` auf dem Parent).

**Fix:** Entweder `group` auf dem Link ergaenzen oder den `group-hover`-Teil gegen ein direktes `hover:translate-x-0.5` auf dem Arrow tauschen. Minimal-Diff:

(Addition: `group ` vor `inline-flex`.) Ausserdem: das `transition-transform` am Arrow fehlt — `group-hover:translate-x-0.5` allein ohne `transition-transform` springt hart. In Offering wird das korrekt kombiniert (siehe `offering-section.tsx:183`).

### WR-03: Inkonsistente React-Keys — Index-based Keys in FAQ + Preview-Graphics trotz stabiler Daten

**File:**
- `apps/website/components/sections/kurz-faq-section.tsx:131` (`key={i}`)
- `apps/website/components/sections/offering-section.tsx:237,265,297` (`key={i}`)
- vs. korrekt: `community-preview-section.tsx:114,133` (`key={article.title}`), `tool-showcase-section.tsx:180` (`key={\`${t.slug}-${i}\`}`)

**Issue:** Community-Preview und Tool-Showcase nutzen bewusst stabile Keys (title / slug). Kurz-FAQ und Offering-Preview-Visualisierungen (CommunityPreview, ToolsPreview, EventsPreview) fallen auf `key={i}` zurueck, obwohl die Daten static + eindeutig sind. Bei React + `AnimatePresence` im FAQ-Fall ist das besonders heikel: Wenn jemand spaeter Fragen reordered, faengt sich der geoeffnete/geschlossene State an den falschen Index. State-Bug-Risiko Low, aber Klasse "Schlechte React-Hygiene, trifft dich bei zukuenftiger Aenderung".

**Fix:** Stabile Keys ableiten:

```tsx
// kurz-faq-section.tsx:131 — statt key={i}:
<li key={item.q} ...>

// offering-section.tsx:237,265,297 — z.B. fuer ToolsPreview:
<div key={t.letter} ...>   // letters sind unique in der Liste
// oder CommunityPreview:
<div key={m.text} ...>
// oder EventsPreview:
<div key={`${e.day}-${e.month}`} ...>
```

Der Verhalten-Unterschied wird erst bei reordering sichtbar — aber weil die Sections schon woanders stabile Keys nutzen, ist die Inkonsistenz an sich ein Code-Smell.

### IN-01: `fadeIn` Motion-Variant ist 6x nahezu identisch dupliziert

**File:**
- `final-cta-section.tsx:47-54`
- `kurz-faq-section.tsx:79-86`
- `offering-section.tsx:75-82`
- `problem-block-section.tsx:75-82`
- `tool-showcase-section.tsx:75-82`
- `trust-section.tsx:69-76`

**Issue:** Exakt dasselbe Pattern (`prefersReducedMotion ? {} : { initial, whileInView, viewport: { once: true, margin: "-10% 0px" }, transition: { duration: 0.6, ease: [0.16, 1, 0.3, 1] } }`) ist in 6 Dateien copy-pasted. Aenderung an der Animation (z.B. Ease-Kurve, Duration) erfordert 6 Updates. Direkter DRY-Kandidat.

**Fix:** In `components/ui/motion-presets.ts` (oder aehnlich) extrahieren:

```ts
// components/ui/motion-presets.ts
export const DS_EASE_OUT = [0.16, 1, 0.3, 1] as const

export function useFadeInUp(opts: { distance?: number; duration?: number; margin?: string } = {}) {
  const prefersReducedMotion = useReducedMotion()
  if (prefersReducedMotion) return {}
  return {
    initial: { opacity: 0, y: opts.distance ?? 16 },
    whileInView: { opacity: 1, y: 0 },
    viewport: { once: true, margin: opts.margin ?? "-10% 0px" },
    transition: { duration: opts.duration ?? 0.6, ease: DS_EASE_OUT },
  } as const
}
```

Dann in den Sections: `const fadeIn = useFadeInUp()`. Spart ~60 Zeilen Duplikation und zentralisiert das Motion-Token.

### IN-04: `onMouseEnter`/`onMouseLeave` als inline `e.currentTarget.style`-Mutation

**File:** `final-cta-section.tsx:179-184`
**Issue:** Der Secondary-CTA-Button mutiert `borderColor` direkt via JS:

```tsx

onMouseLeave={(e) => (e.currentTarget.style.borderColor = "var(--border)")}
```

Das ist direkter DOM-Zugriff, der mit der CSS `hover:`-Pseudoklasse trivial ersetzt werden kann. Funktional aequivalent, aber laeuft in React's render cycle und kollidiert potenziell mit `transition-colors`. **Fix:**

```tsx
<a
  href="https://tools.generation-ai.org"
  target="_blank"
  rel="noopener noreferrer"
  className="group inline-flex items-center ... border border-[var(--border)] hover:border-[var(--border-accent)] transition-colors duration-300 ease-[cubic-bezier(0.16,1,0.3,1)] ..."
>
```

Inline `borderColor` + die `style={{ borderColor: "var(--border)" }}`-Zeile fallen weg.

### IN-06: `setTimeout` in `home-client.tsx` ohne Cleanup

**File:** `apps/website/components/home-client.tsx:26-29`

```tsx
const handleSplashComplete = () => {
  setSplashDone(true)
  setTimeout(() => setShowContent(true), 50)
}
```

**Issue:** Wenn die Komponente unmounted wird, waehrend der 50ms-Timer noch laeuft, ruft er `setShowContent` auf einer unmounted component auf. React 18/19 warnt da nicht mehr laut, aber best practice ist, die Timer-ID zu tracken und im cleanup zu clearen — oder diese Kaskade ueber `useEffect` zu loesen. Praktisch ist das Risiko im HomeClient-Fall nahe null (Komponente laeuft fuer die gesamte Page-Lifetime), daher Info-Level.

**Fix:** Optional `useEffect`-Kaskade:

```tsx
useEffect(() => {
  if (!splashDone) return
  const t = setTimeout(() => setShowContent(true), 50)
  return () => clearTimeout(t)
}, [splashDone])

const handleSplashComplete = () => setSplashDone(true)
```

### IN-08: `tabindex="-1"` auf geclonten Marquee-Kindern koennte fokussierbare Nachfahren nicht ausschliessen

**File:**
- `tool-showcase-section.tsx:157-159`
- `trust-section.tsx:140-144`

```tsx
const c = el.cloneNode(true) as HTMLElement
c.setAttribute("aria-hidden", "true")
c.setAttribute("tabindex", "-1")
```

**Issue:** `tabindex="-1"` am aeusseren `<li>` entfernt das `<li>` selbst aus der Tab-Order, aber die inneren `<a>`-Elemente bleiben fokussierbar. In Tool-Showcase z.B. sind die Clones vom `<li>` her `tabindex=-1`, aber der `<a>` innen behaelt seine natuerliche tabbability. Praktisch heisst das: Keyboard-User durchlaufen jedes Tool **zweimal** (Original + Clone), was die Aria-Hide-Intention unterlaeuft.

**Fix:** Beim Cloning auch alle fokussierbaren Nachkommen `inert` machen (besser) oder deren `tabindex` auf `-1` setzen:

```tsx
const c = el.cloneNode(true) as HTMLElement
c.setAttribute("aria-hidden", "true")
c.setAttribute("inert", "")  // supported in alle modernen Browser 2024+
// oder als Fallback:
c.querySelectorAll<HTMLElement>("a, button").forEach((n) => n.setAttribute("tabindex", "-1"))
track.appendChild(c)
```

`inert` ist die sauberere Variante; es entfernt alle Nachfahren aus der Tab-Order, macht sie nicht fokussierbar und blendet sie fuer Screen-Reader aus — genau das, was der Marquee-Clone will.
