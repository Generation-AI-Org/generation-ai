# Phase 22.8 Plan 01: DS-Compliance Audit Summary

**Read-only DS-compliance audit identifying 24 findings (6 must / 11 should / 7 nice) across apps/website, apps/tools-app, packages/ui, packages/emails — clustered into 7 actionable fix-types for Plan 22.8-02 to consume.**

## Outcome

Produced `22.8-01-AUDIT.md` with a per-finding decision table (file:line, severity, recommendation). The audit confirms the codebase is **mostly DS-compliant** — heavy correct token usage — with drift concentrated in 5 cluster areas:

1. Card-hover-pattern divergence (3 competing languages: `scale(1.015)` canonical, `translate-y-[2-3px]` website-cards, `translate-y-0.5` portrait-cards)
2. Hardcoded Tailwind status-colors in tools-app auth/settings flows (light-mode-broken)
3. Logo-hover inconsistency (3 callsites, 3 different patterns)
4. Border-radius drift (4 buttons using `rounded-md/lg` instead of canonical `rounded-full`)
5. Inline JS-driven hover handlers in 2 form-submit-buttons

The audit also flags 1 token-source bug: `--color-destructive` in `apps/website/app/globals.css` is hardcoded `#DC2626` instead of referencing the existing `--status-error` token, which would prevent silent light-mode drift if DS later adjusts.

- **Files scanned:** ~120 components via grep, ~15 read in full
- **Excluded:** `node_modules`, `.next`, `__tests__`, `*.spec.*`, `*.test.*`
