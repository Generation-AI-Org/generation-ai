# Plan 22.8-01 — DS-Compliance Audit Report

> **Read-only Audit.** Inventory of all DS-compliance gaps in `apps/website/`, `apps/tools-app/`, `packages/ui/`, `packages/emails/`. Output: per-finding decision table with file:line, severity, and recommended fix. NO code changes made — those happen in Plan 22.8-02 after Luca-Review.

## Executive Summary

The codebase is **mostly DS-compliant** — heavy use of `var(--…)` tokens, brand-blue/-pink/-neon scales, slate-N neutrals. The drift falls into 5 clusters:

1. **Card-Hover-Pattern divergence** — three competing languages: canonical `scale(1.015)` (DS-spec, used by tools-app ContentCard), `translate-y-[2px/3px]` (most website cards), and `translate-y-0.5` (about/partner/team cards). DS-spec lines 89: canonical = `scale(1.015) + Shadow + Border-fade-to-accent`.
2. **Hardcoded Tailwind status colors** — `bg-red-500`, `text-red-400`, `text-green-400`, `bg-emerald-500`, `border-red-500/30` for error/success states instead of `--status-error` / `--status-success` tokens. Concentrated in tools-app auth/settings flows. **Light-mode broken** in these spots — `red-500` is dark-only by design.
3. **Logo-Hover inconsistency** — Logo component (`packages/ui/Logo.tsx`) has NO hover-state. Three call-sites apply different ad-hoc hovers: header `hover:opacity-90 transition-opacity`, footer/tools-app/login: NO hover. No DS-canonical Logo-Hover defined.
4. **Border-radius drift** — Buttons rendered with `rounded-md`, `rounded-lg`, and `rounded-full` mixed across the codebase. DS-spec lines 83+99: Buttons = `rounded-full` (pill), Cards/Modals/Inputs = `rounded-2xl`.
5. **Inline JS-driven hover-handlers in Forms** — `JoinFormCard` + `partner-contact-form` use `onMouseEnter/Leave` to set `style.transform = 'scale(1.03)'` instead of CSS `hover:` classes. Bypasses `prefers-reduced-motion`, makes pattern-grep impossible, breaks `:active` states.

**No threat-flag findings.** No new attack surface, no auth/security regressions in scope. This audit is purely visual/DS-compliance.

## Severity Definitions

- **must** — Pro-Launch blocker. Visual broken in one or both themes, OR DS-canonical violation that propagates across many sites if untouched.
- **should** — Real DS drift. Fixable in 22.8-02 with ≤30min per finding. Not visually broken, just inconsistent.
- **nice** — Cosmetic / nuance. Can go to backlog.

4. **A11y / focus-rings / contrast** — completely out of scope. Saw mixed `focus-visible:outline` vs `focus-visible:ring` patterns; saw 7+ different focus-style implementations. Different audit, different phase.
5. **Legacy welcome-banner** (`apps/website/components/welcome/community-banner.tsx`) uses old shadcn-button-style instead of any of the canonical patterns. May be dead code worth investigating in Plan 22.8-02 scope-decision (delete vs. fix).
