# Plan 22.8-06 — Join Status-Switch + Studienfeld + Schema-Migration

## Goal

`/join` soll nicht mehr implizit nur Studierende erfassen. Der Flow bekommt einen Status-Switch (Studierend / Berufstätig / Alumni / Sonstiges), ein strukturiertes Studienfeld und eine additive Waitlist-Migration.

## Changes

- Neue Supabase-Migration: `waitlist_status` ENUM, `waitlist.status`, `waitlist.study_field`, Backfill auf `student`, `waitlist.university` nullable.
- `JoinFormCard`: Status-Switch als Radio-Button-Grid, conditional Hochschule/Kontext-Feld, strukturierter Studienfeld-Select, SessionStorage-Draft erweitert.
- Waitlist-V1 und Signup-V2 validieren denselben Status-Wertebereich (`student | working | alumni | other`).
- Signup-Metadata schreibt `status`, optional `university`, `study_field`, `study_program`.
- E2E-Join-Spec um Status-Switch und Studienfeld erweitert.

## Decisions

- `study_program` bleibt als Legacy-Feld bestehen und wird aus `study_field` mitgefüllt, damit bestehende Consumers nicht sofort brechen.
- Die Migration wird nur als Repo-SQL geliefert. Live-Anwendung auf Supabase bleibt ein expliziter Approval-Schritt.
- `working` blendet Hochschule aus; `student` und `alumni` verlangen Hochschule weiterhin auf App-Ebene.

## Validation

- `pnpm --filter @genai/website exec tsc --noEmit` grün.
- `pnpm build` grün; `/join` bleibt `ƒ` dynamic.
- Supabase-Migration via MCP angewendet und Schema verifiziert (`status`, `study_field`, `university nullable`).
- `E2E_BASE_URL=http://localhost:3000 pnpm --filter @genai/e2e-tools exec playwright test tests/join.spec.ts`: 12/12 grün.

## Must-Haves

- [x] Waitlist-V1 und Signup-V2 akzeptieren dieselbe neue Datenform.
