# Plan 24-04 — Simple Widgets

## Objective

Build the 4 non-drag-drop widgets: `<CardPickWidget>` (W1), `<MCWidget>` (W9 — reuses CardPick layout in single-column), `<ConfidenceSliderWidget>` (W7 using shadcn Slider), and `<FillInWidget>` (W8 inline dropdown in code block). Each is keyboard-operable and A11y-compliant per UI-SPEC.

## Context

Per 24-UI-SPEC W1/W9/W7/W8 sections: these widgets share a lift-up `onAnswer` callback, expose `isAnswered: boolean` for the layout's "Nächste Aufgabe" enable-gate, and use `role="radiogroup"` / `role="slider"` / native `<select>` patterns. No shadcn block install needed beyond `<Slider>` which is shadcn official.

  <action>
    Create `apps/website/components/test/widgets/fill-in-widget.tsx`:
    - Client component
    - Props: `WidgetProps<FillQuestion, FillAnswer>`
    - Layout: code block using `<pre className="bg-[var(--slate-2)] rounded-xl p-4 font-mono text-sm leading-relaxed overflow-x-auto" data-widget-type="fill">`
    - Rendering: split `codeTemplate` by `{{BLANK_ID}}` tokens. For each segment render plain text; for each blank token, render a `<select>`:
      ```tsx
      <select
        className={cn(
          'inline-block mx-1 bg-[var(--slate-3)] border border-[var(--slate-6)] rounded px-2 py-0.5 text-sm font-mono text-[var(--text)]',
          'focus:border-[var(--border-accent)] focus:outline-none',
          'disabled:opacity-60'
        )}
        disabled={disabled}
        value={selections[blank.id] ?? ''}
        aria-label={`Parameter auswählen für ${blank.id}`}
        onChange={e => updateSelection(blank.id, e.target.value)}
      >
        <option value="">— wählen —</option>
        {blank.options.map(opt => <option key={opt.value} value={opt.value}>{opt.value}</option>)}
      </select>
      ```
    - `updateSelection` merges the blank into the answer: `onAnswer({ questionId, type: 'fill', selections: { ...selections, [id]: value } })`
    - Token-splitting util: use `codeTemplate.split(/\{\{([A-Z_0-9]+)\}\}/g)` — alternating text/token segments.
    - Shiki rendering is NOT applied at this layer (Plan 24-06 shows shiki pattern for best-prompt). For fill-in, plain `<pre>` with `font-mono` is acceptable per UI-SPEC (readability over syntax color when form controls are embedded).
  </action>
