<objective>
Refactor Tool-Showcase and Community-Preview sections from `'use client'` stubs to **async Server Components** that fetch their data live, then re-architect the landing page boundary so a client `home-client.tsx` can host server-rendered children.

**Why this is the most subtle plan in the phase:** `home-client.tsx` is `'use client'` (it uses motion + state + TerminalSplash), so it cannot import async Server Components directly — React forbids this. The fix is the standard "server-pass-through" pattern: `app/page.tsx` (Server Component) renders the two server sections and passes the rendered ReactNodes as **props** to `<HomeClient />`. HomeClient slots them in by-position with the rest of the page.

Purpose: Block B's payoff. Tool-Showcase no longer fakes data — pulls from tools-app API. Community-Preview's article column reads real MDX from Plan 26-01 (D-08 Option A). Events column stays stub (D-21) — explicit Phase 22.5 territory.

Output: Landing page with two genuinely-data-driven sections, end-to-end live data path, fallback stubs in place if API fails.

Implements: D-08 (Option A — MDX teaser), D-15 (in-place upgrade with ISR + fallback), D-21 (events stays stub).

**Has `checkpoint:human-verify` because the landing rewires data sources — visual regression risk + mobile responsiveness needs eyeballs.**
