# Phase 22.5 — `/events` Seite

> Events als **Members-Only-Akquisitionshebel**. Öffentlich sichtbare Liste kommender Events, aber Anmeldung gated hinter Signup. MDX-basierte Events-Pipeline im Repo, kein Circle-API-Pull. Umsetzung nach Simons Konzept §5.

---

## Strategische Klärung

Simons Konzept §5.6 ist eindeutig: **„Unsere Events sind exklusiv für Mitglieder."** Events sind **Akquisitionshebel** für Signup, nicht Public-Discovery-Ziel. Das vereinfacht die Architektur radikal:

- Keine Circle-Public-Events-Experimente nötig
- Keine Circle-API-Integration nötig
- Kein Luma-Circle-Hybrid nötig
- Event-Hosting bleibt flexibel pro Event (Circle-internal, Luma, whatever)
- Unsere `/events`-Seite ist nur der **Discovery- und Conversion-Kanal** auf der Hauptdomain

---

## Mission

Besucher:innen sehen auf öffentlicher `/events`-Seite welche Events anstehen (Titel, Datum, Format, Level, Ort, Kurzbeschreibung). Klick auf „Anmelden" im Modal leitet bei Non-Members in den Signup-Flow um (Phase 23 `/join`), bei Members direkt auf die Event-URL (Circle internal, Luma, etc.).

Archiv zeigt vergangene Events minimal als Social Proof („das haben wir schon gemacht") — kein Modal, keine Details, reiner Track-Record.

---

## Scope

**In-scope:**
- Hero-Sektion nach Konzept §5.3: Section-Header „Generation AI · Events" + H1 „Events, die dich weiterbringen." + Subhead
- **Kommende Events** als 3-Kachel-Grid
  - Metadaten pro Kachel: Titel, Datum+Uhrzeit, Format-Tag (Workshop/Speaker/Masterclass), Level-Tag (Einsteiger/Fortgeschritten/Expert), Ort (Online/Stadt/hybrid), Partner (nur bei Masterclasses), Anmelde-Button
  - Bei >3 Events: „Mehr Events anzeigen ↓" Button (auf gleicher Seite ausklappen)
- **Event-Modal** bei Kachel-Klick
  - Längere Beschreibung, Agenda, Speaker-Info
  - Anmelde-Button
  - Optional: ICS-Download (Kalender-Export)
- **Anmelde-Flow:**
  - Klick „Anmelden" → Auth-Check
    - Logged in → Redirect zur `ctaUrl` des Events (Circle/Luma/etc.)
    - Not logged in → Redirect zu `/join?redirect_after=/events/[slug]` (Signup-Gate)
- **Formate-Sektion** §5.5: 3 Kacheln (Workshops / Speaker Sessions / Masterclasses) mit Kurzbeschreibungen
- **Members-Only-Hinweis** §5.6: explizite Sektion erklärt warum Signup-Gate („Kostenlos, 2 Minuten")
- **Archiv-Sektion** §5.7: minimale Kacheln (Foto + Titel + Datum + 2-Satz-Recap), keine Modals
- **Abschluss-CTA** §5.8: „Keine Events verpassen. [Jetzt beitreten]"
- **MDX-Pipeline** in `apps/website/content/events/`:
  ```
  apps/website/content/events/
  ├── 2026-05-15-prompting-masterclass.mdx
  ├── 2026-05-28-agents-workshop.mdx
  └── past/
      └── 2026-03-10-launch-event.mdx  (Archiv)
  ```
- Frontmatter-Schema pro Event:
  ```yaml
  ---
  title: string
  date: ISO-DateTime
  format: "Workshop" | "Speaker Session" | "Masterclass"
  level: "Einsteiger" | "Fortgeschritten" | "Expert"
  location: "Online" | string (Stadt) | "hybrid"
  partner: string | null    # nur bei Masterclasses
  ctaUrl: string            # Circle-Event-URL oder Luma-URL
  slug: string
  speakers: Array<{name, role, avatar?}>  # optional
  ---
  Body (Markdown mit Beschreibung, Agenda)
  ```
- Build-Logic: Events mit `date >= now` → „Kommende Events", `date < now` → Archiv
- Event-Detail-URL: `/events/[slug]` (Modal-Content als Standalone-Page rendered für Deep-Linking + SEO)

**Out-of-scope:**
- Circle-API-Integration (deferred — aktuell nicht nötig)
- Luma-Integration (nur als Link-Target in `ctaUrl`, keine Embed)
- ICS-Download in V1 (nur wenn trivial, sonst Roadmap)
- Event-Filter („nach Format", „nach Level") — erst ab 5+ parallelen Events nötig
- Search auf Events (Roadmap)
- Analytics pro Event-Klick (Roadmap)
- Email-Benachrichtigungen bei neuen Events (Roadmap — kommt als Feature der Community-Plattform)
- Finales Wording — Placeholder, Copy-Pass in Phase 27

## Decisions

- **D-01** — Events sind members-only per Strategie (Simon §5.6). Die Seite selbst ist public, Anmeldung ist gated.
- **D-02** — MDX-Files als Content-Source. Parallel-Pipeline zu `/community`-Artikel (Phase 26), gleiche Mechanik. Kein CMS, keine API.
- **D-03** — Event-Hosting bleibt flexibel. `ctaUrl` im Frontmatter zeigt auf Circle-Event, Luma-Event, oder anderen Dienst. Kein Lock-in auf eine Plattform.
- **D-04** — Modal zeigt Event-Details öffentlich. Nur Anmeldung ist gegated. Argument: Unentschlossene müssen das Event-Detail sehen können, um Signup-Motivation zu bekommen.
- **D-05** — Event-Detail als eigene URL `/events/[slug]` zusätzlich zum Modal. Grund: SEO + Share-Links funktionieren. Modal ist UX-Preview, URL ist Permalink.
- **D-06** — Archiv ohne Modal. Reiner Social Proof. Pflegeaufwand minimal. 2-Satz-Recap + Foto reicht.
- **D-07** — Anmelde-Flow: Auth-Check Client-side oder Server-side? → Empfehlung Server-Component-Check bei Button-Click (Route-Handler), Redirect entsprechend. Alternative: Client-side mit Supabase-Session + `useRouter`. TBD im Plan.
- **D-08** — Signup-Gate nutzt Query-Param `?redirect_after=/events/[slug]`, damit Nach-Signup direkt zum Event zurückgeleitet wird. Integration mit Phase 23 `/join` Flow.
- **D-09** — „Mehr Events anzeigen ↓" erweitert das Grid auf gleicher Seite (Client-State), nicht separate Unterseite. Reduziert Reibung.
