    **Doppelter reduced-motion-Schutz:**
    1. globals.css `@media (prefers-reduced-motion: reduce) { .animate-marquee { animation-play-state: paused } }` (Plan 01)
    2. JS-side `useReducedMotion()` → wenn true, Marquee-Komponente gar nicht erst rendern, statisches Layout zeigen.

    Beide Schichten zusammen erfüllen den Test in landing.spec.ts (`R1.8 — Trust marquee pauses on prefers-reduced-motion`) — entweder ist die Marquee gar nicht im DOM (JS-fallback), oder sie ist im DOM mit `animation-play-state: paused` (CSS-Guard). Der Test prüft auf `paused` → wenn JS-fallback aktiv ist, gibt es kein `.animate-marquee`-Element → Test muss robust sein. Daher: in landing.spec.ts (Plan 01 Task 3) wurde der Test geschrieben als `getComputedStyle(...).animationPlayState === "paused"`. Bei JS-fallback existiert das Element nicht → Test schlägt fehl.

    **Fix für robustere Testbarkeit:** Wir lassen Marquee IMMER im DOM rendern (auch bei reduced-motion), aber stoppen sie zusätzlich via JS:
    ```tsx
    return (
      <Marquee
        pauseOnHover
        className={`[--duration:40s] ${prefersReducedMotion ? "[animation-play-state:paused!]" : ""}`}
      >
        {stubPartners.map((name) => (<PartnerTile key={name} name={name} />))}
      </Marquee>
    )
    ```

    Damit ist `.animate-marquee` immer im DOM, und `animationPlayState` ist bei reduced-motion `paused` (sowohl via globals.css als auch via inline-class). Der Static-Layout-Pfad wird nicht gebraucht — wir entfernen den `prefersReducedMotion ? Static : Marquee` Conditional und nehmen IMMER Marquee.
